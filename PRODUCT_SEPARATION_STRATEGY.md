# 免费体验版 vs 付费 Agent：产品分离策略

> 版本：v1.0 ｜ 日期：2026-07-16
> 配套：AGENT_UPGRADE_ROADMAP.md / AGENT_TECH_STACK.md
> 背景：现有外贸验货应用为免费市场验证工具；升级后的 Agent 为收费产品，需理清两者关系。

---

## 一、核心结论

**应该区分，但采用「共享内核 + 分离产品」，不是做两套重复的东西。**

- 免费版 ≠ 要被淘汰的旧产品
- 免费版 = 付费版的「获客漏斗顶端」+「数据金矿」
- 两者是**漏斗关系**（引流→转化），不是新旧替代关系

---

## 二、两个产品定位

| 维度 | 免费体验版 | 付费 Agent 专业版 |
|------|-----------|------------------|
| 使命 | 市场验证 + 获客引流 | 商业转化、盈利 |
| 目标用户 | 潜在客户、试用者 | 真实付费验货商 |
| 产品形态 | 轻量单轮，够用即可 | 完整 Agent（追问/查标准/记历史）|
| 数据性质 | 可脱敏沉淀为知识库 | 客户真实业务数据，需隔离 |
| 计费 | 免费/限次数 | 订阅/按量付费 |

---

## 三、分离架构：共享内核，三层分离

```
        ┌──────── 共享核心库 core/ ────────┐
        │  AQL判定引擎 · PDF生成 · AI调用封装 │  ← 一份代码，两边复用
        └──────┬──────────────────┬────────┘
               │                  │
      ┌────────▼───────┐  ┌───────▼────────┐
      │  免费体验版      │  │  付费Agent专业版 │
      │  （引流/验证）   │  │  （商业产品）    │
      ├────────────────┤  ├────────────────┤
      │ 独立部署/域名    │  │ 独立部署/域名    │
      │ 独立Supabase    │  │ 独立Supabase    │ ← 数据隔离/合规
      └────────────────┘  └────────────────┘
```

### 第1层：代码层 —— monorepo + 抽公共库
- 把 AQL 引擎、PDF 生成、AI 调用封装抽成 `core/` 共享模块
- 两个 app 目录（`app_free/`、`app_pro/`）各自引用 core
- 改一次内核两边都受益，杜绝重复维护
- 目录建议：
  ```
  repo/
  ├── core/                 # 共享：AQL引擎、PDF、AI封装、工具函数
  ├── app_free/             # 免费体验版（现有单轮逻辑）
  ├── app_pro/              # 付费Agent版（多轮+工具+记忆）
  └── requirements.txt
  ```

### 第2层：部署/品牌层 —— 完全分开
- 两个独立域名、两个 Streamlit 实例
- 付费版有独立品牌、定价页、专业形象，不被"免费 MVP"调性拖累
- 各自独立迭代、独立发版，互不影响

### 第3层：数据层 —— 两个独立 Supabase 项目
- 合规与安全：付费客户真实业务数据必须与免费试用数据隔离
- 计费系统各自独立（免费版计次数；付费版接订阅/支付）
- 避免免费版的问题（如临时邮箱、刷量）污染付费数据

---

## 四、免费版的隐藏价值：数据反哺

- 免费版收集的验货数据（**必须脱敏**）→ 各品类常见缺陷模式
- 这些成为付费版 Agent **记忆层的冷启动语料**
- 直接解决升级路线"阶段三记忆层无数据冷启动"的难题
- 即：免费版一边验证市场，一边帮付费版攒知识库

> 注意：数据反哺需严格脱敏 + 合规（去除客户身份/商业敏感信息），仅提取通用缺陷模式。

---

## 五、商业衔接：漏斗打通

```
免费体验版  ──「升级专业版」CTA──▶  付费Agent专业版
（试用满意）                        （订阅转化）
```

- 免费版内置转化入口（如"解锁完整 AI 验货员"）
- 试用满意的用户自然流向付费版
- 区分开，但用漏斗连起来，形成获客→转化闭环

---

## 六、落地顺序建议

1. **先抽 core/**：把现有 AQL/PDF/AI 逻辑抽成共享库（现有应用改为引用 core，行为不变）
2. **复制出 app_pro/**：基于 core 起新的付费版骨架
3. **在 app_pro 做 Agent 升级**：按 ROADMAP 阶段一/二/三推进
4. **免费版保持稳定**：仅做引流入口和数据脱敏采集的小改动
5. **打通漏斗**：免费版加升级 CTA，付费版接支付

---

## 七、要避开的两个坑

| 坑 | 后果 | 规避 |
|----|------|------|
| 两套完全独立、代码重复 | 维护成本翻倍、内核修复要改两遍 | 抽 core 共享库 |
| 免费付费混在一个应用/一个库 | 免费版问题污染付费数据；商业形象受损；合规风险 | 部署/品牌/数据三层分离 |

---

## 八、一句话总结

**共享一个内核（core），长出两个产品：免费版负责引流和攒数据，付费版负责赚钱和记客户。用漏斗连接，用数据隔离，各自独立演进。**
