# 验货 AI → 验货 Agent 升级路线图

> 版本：v1.0 ｜ 日期：2026-07-13
> 适用项目：inspection-agent-mvp（Streamlit + qwen-vl + Supabase）

---

## 一、现状定位（升级的起点）

**当前本质：单轮 AI 功能**，不是 Agent。

```
上传图片 → 一次 qwen-vl 调用 → 返回 JSON → Python 判定 AQL → 生成 PDF
```

| 维度 | 现状 | Agent 应具备 |
|------|------|-------------|
| 交互 | 单轮，一次出结论 | 多轮对话、主动追问 |
| 决策 | 流程写死 | 自主判断下一步做什么 |
| 工具 | 无（AQL 表硬编码在代码里） | 动态调用工具箱 |
| 记忆 | 无（刷新即丢） | 客户/工厂历史记忆 |
| 数据 | 仅当次图片 | 检索历史缺陷、标准库 |

**好消息**：核心资产已具备——AQL 判定引擎（已按 ANSI Z1.4 修正）、Supabase 数据层、鉴权/计费、PDF 生成。升级是"加能力"，不是"推翻重来"。

---

## 二、三阶段路线总览

```
阶段一：多轮追问        阶段二：工具调用          阶段三：记忆 + 自主
（对话式 Agent）        （核心，真正的 Agent）    （长期价值）
   1-2 天                  3-5 天                   持续迭代
   ↓                       ↓                        ↓
让 AI 会问              让 AI 会查                让 AI 会记
```

**优先级建议（按用户痛点）：**
- 用户抱怨"判断不准/不专业" → 先做**阶段二**（准确度提升最明显）
- 用户抱怨"交互生硬/不好用" → 先做**阶段一**
- 有回头客/固定工厂客户 → 尽早做**阶段三**（差异化护城河）

---

## 三、阶段一：多轮追问（对话式 Agent）

### 目标
AI 发现信息不足时**主动追问**，而不是硬着头皮出结论。

### 典型场景
- 图片模糊/角度不够 → "请补拍产品底部的合规标签"
- 缺关键信息 → "这批货的订单数量是多少？验货标准按哪个 AQL？"
- 缺陷存疑 → "图3 这个位置看起来像划痕也像反光，能否换角度再拍一张？"

### 技术改动
1. `analyze_product_images()` 从"单次 create"改为**带 messages 历史的多轮循环**
2. 新增返回类型：`{"action": "ask", "question": "..."}` vs `{"action": "report", ...}`
3. Streamlit 侧用 `st.chat_message` / `st.session_state["chat_history"]` 承载对话
4. system prompt 增加规则："信息不足时先追问，不要臆断"

### 验收标准
- [ ] 上传模糊图片时，AI 会要求补拍而非瞎判
- [ ] 未填订单数量时，AI 会主动询问
- [ ] 追问 ≤2 轮内收敛到出报告，不无限循环
- [ ] 保留"跳过追问，直接出报告"的逃生出口

### 风险与规避
- **风险**：多轮增加 token 消耗和等待时间 → 限制最多 2 轮追问
- **风险**：老用户不适应对话式 → 保留"快速模式"开关

## 四、阶段二：工具调用（核心里程碑）

### 目标
给 AI 配"工具箱"，让它像真验货员一样**边想边查**，而不是凭记忆编。

### 工具清单（优先级排序）

| 工具 | 作用 | 数据来源 | 优先级 |
|------|------|---------|--------|
| `query_aql_plan` | 按订单量+AQL 查真实抽样方案 Ac/Re | 已有 `get_aql_sample_info()` | P0 |
| `search_defect_history` | 检索同类产品历史缺陷模式 | Supabase `inspection_reports` | P0 |
| `lookup_standard` | 查国标/行业验货标准要点 | 标准知识库（可先内置常见品类） | P1 |
| `calc_sampling` | 计算/校验抽样数量 | 已有逻辑 | P1 |
| `get_product_profile` | 查该产品/工厂的历史档案 | Supabase | P2 |

### 技术改动
1. 用 OpenAI-compatible **function calling / tools** 参数（qwen 支持）
2. 定义 tools schema，实现工具执行层（Python 函数 → 返回 JSON）
3. Agent 主循环：`模型请求工具 → 执行 → 回填结果 → 再推理`，直到产出最终报告
4. 把现有硬编码逻辑（AQL 表、判定）封装成工具，AI 主动调用而非写死在 prompt

### 关键设计：为什么这样最值
现在 AQL 判定虽然已在 Python 层修正，但 **AI 看图时并不"知道"完整标准**。工具调用让 AI 能：
- 先问清订单量 → 调 `query_aql_plan` 拿到真实 Ac/Re → 有依据地数缺陷
- 调 `search_defect_history` → "这类打火机历史上 80% 问题在点火装置，重点看这里"
- 准确度和专业度是质的提升，而非 prompt 堆砌

### 验收标准
- [ ] AI 能自主决定调用哪个工具、调用顺序合理
- [ ] `query_aql_plan` 返回结果被正确用于判定
- [ ] `search_defect_history` 能召回同品类历史缺陷并影响检查重点
- [ ] 工具调用失败有兜底（降级为无工具模式）
- [ ] 单次验货工具调用总数有上限（防止死循环烧 token）

### 风险与规避
- **风险**：工具调用链过长、延迟高 → 设最大 3-5 步；关键工具并行调用
- **风险**：模型幻觉调用不存在的工具 → 严格 schema 校验 + 白名单
- **风险**：qwen function calling 稳定性 → 加重试 + 结构化输出兜底

## 五、阶段三：记忆 + 自主任务（长期护城河）

### 目标
Agent 记住客户/工厂/产品的历史，形成越用越准的飞轮。

### 能力
1. **客户记忆**：记住某工厂的历史质量问题
   - "这家工厂上次就是走线问题，这次重点看走线"
2. **产品档案**：同一 SKU 多次验货，趋势分析
   - "该产品次品率从上月 3% 升到本月 7%，建议预警"
3. **主动任务**：
   - 定期汇总某客户的质量趋势报告
   - 发现异常主动提醒（结合 cron）

### 技术改动
1. Supabase 增加记忆表：`factory_memory` / `product_profile`
2. 检索增强：验货前先拉取相关历史注入上下文
3. （可选）向量检索：缺陷描述语义相似度匹配
4. 定时任务：周期性生成客户质量趋势报告

### 验收标准
- [ ] 同一工厂第二次验货时，AI 引用了上次的问题
- [ ] 产品多次验货能给出趋势判断
- [ ] 记忆不串号（工厂 A 的记忆不泄漏给工厂 B）

### 风险与规避
- **风险**：记忆数据隐私/串号 → 严格按 user_id + factory_id 隔离
- **风险**：历史数据不足冷启动 → 前期靠品类通用知识兜底

## 八、现有网页产品如何保留（重要）

**结论：现有产品 100% 保留，升级是"加楼层"不是"拆地基"。**

### 共用同一套底座（无一丢弃）
| 现有资产 | 升级后 | 变化 |
|---|---|---|
| Streamlit 网页 | 继续用，加对话区 | 增量 |
| Supabase 数据/鉴权/计费 | 原样复用 | 无 |
| AQL 判定引擎 | 变成 Agent 的一个工具 | 逻辑不动 |
| PDF 生成 | 原样复用 | 无 |
| 现有用户/次数/报告 | 完全不受影响 | 无 |

### 落地策略：双模式并存 + 灰度
网页内加模式开关：
- **⚡ 快速模式**（现有单轮）：默认保留，老用户零感知
- **🤖 智能模式**（新 Agent 对话）：可灰度开放，验证稳定再全量

优势：
1. 老用户习惯不变（传图直接出报告照旧）
2. 新功能可灰度、可分批放量
3. **随时可回退**——Agent 出问题关掉开关即回到现稳定态（安全垫）
4. 同一代码库，工具层两模式共用，不维护两套系统

### 技术共存方式
- `analyze_product_images()`（单轮）保留不动 = 快速模式
- 新增 `run_agent()`（多轮循环）= 智能模式
- 两者共用同一套工具函数、Supabase、PDF 生成
- Streamlit 用 `st.session_state["mode"]` 切换渲染
